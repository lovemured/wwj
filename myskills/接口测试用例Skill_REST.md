# 接口测试用例 Skill（REST/HTTP）—标准版


> 目标：提供一套可复用的**接口测试用例设计/编写规范与模板**，覆盖功能&异常、安全、性能/压测、兼容/稳定性四大维度，帮助团队在评审、落地与自动化时保持一致性与可追溯性。

---

## 1. 适用范围与假设

**适用**：REST/HTTP 接口（JSON 为主，亦适用于表单/文件上传等）。  
**不适用**：WebSocket/GraphQL（需另行补充协议特有用例）。  
**默认假设**：
- 接口有清晰的契约（OpenAPI/Swagger、API 文档或网关配置任一即可）。
- 具备至少一个可测试环境（DEV/SIT/UAT/预发）和可获取的访问方式（域名/网关/白名单）。

---

## 2. 测试目标（你要“证明”什么）

接口测试用例的目标不是“把接口调通”，而是证明接口在**不同输入、不同上下文、不同负载**下满足：

1) **功能正确**：符合业务规则与接口契约；  
2) **异常可控**：错误码/错误信息/可恢复性符合约定；  
3) **安全合规**：鉴权、越权、注入、敏感信息等风险被控制；  
4) **性能达标**：在目标并发/吞吐下响应时间、错误率可接受；  
5) **稳定可靠**：幂等、重试、超时、限流、版本兼容等机制有效。

---

## 3. 前置准备（编写用例前必须拿到的信息）

> 如果缺失其中任意一项，会导致用例难以“可执行/可验收”。

### 3.1 契约信息（至少一种）
- OpenAPI/Swagger（优先）
- API 文档（参数、字段、枚举、示例）
- 网关/服务端代码中的接口定义（仅在文档缺失时）

### 3.2 环境与依赖
- 测试环境地址、网关路径、鉴权方式（Token/JWT/签名/AKSK/Session）
- 依赖服务说明（需要 Mock 还是可直连）
- 数据准备方式（后台造数、DB 脚本、预置账号、测试数据接口等）

### 3.3 口径与阈值（性能/稳定性必须明确）
- SLA：P95/P99 响应时间、可接受错误率、目标 QPS/并发
- 超时、重试、限流策略（客户端/网关/服务端分别是什么）
- 幂等策略（幂等键、请求去重窗口、重复提交处理）

---

## 4. 用例设计方法（推荐组合拳）

### 4.1 等价类 + 边界值（参数层）
对每个字段至少覆盖：
- **必填/非必填**：缺失、为空字符串、为 `null`（如允许）  
- **类型**：string/number/boolean/object/array 类型错误  
- **范围**：最小值-1、最小值、最大值、最大值+1（数字/长度/分页）  
- **格式**：日期/邮箱/手机号/UUID/正则不匹配  
- **枚举**：合法枚举、非法枚举、大小写（如不区分）  
- **数组/列表**：空数组、超长数组、重复元素、顺序敏感/不敏感

### 4.2 决策表/因果图（业务规则层）
当业务规则由多个条件组合触发（例如：会员等级 + 是否新用户 + 活动开关）时，用**决策表**列出条件组合与预期结果，减少漏测。

### 4.3 状态迁移（流程层）
对“有状态资源”尤其重要（订单、任务、审批流）：
- 正常流：A→B→C  
- 异常流：A→失败→重试→B  
- 非法流：绕过中间状态、重复提交同状态

### 4.4 风险驱动（质量属性层）
对高风险接口（支付、登录、权限、导出、批量操作）额外加大：
- 安全：越权/注入/敏感信息/重放  
- 性能：大字段、大列表、热点 Key  
- 稳定性：幂等、限流、降级、超时

---

## 5. 用例分层模型（建议每个接口至少具备）

> 实操中可按“必须 / 建议 / 视情况”划分优先级。

### 5.1 基础层（必须）
- 正向：典型输入成功返回
- 参数校验：必填、类型、范围、格式
- 认证：无 Token / 过期 Token / 非法 Token
- 返回结构：字段存在性、类型、枚举、可空性
- 幂等（若声明支持）：重复请求一致性

### 5.2 业务层（必须）
- 关键业务规则（限额、库存、状态流转、去重）
- 关键数据一致性校验（写 DB/发消息/更新缓存）

### 5.3 异常与容错层（必须）
- 下游失败/超时（若可模拟）：返回码、重试、降级文案
- 并发冲突：重复提交/并发更新（乐观锁、版本号、ETag）

### 5.4 安全层（建议）
- 越权：水平越权（他人资源）、垂直越权（角色权限）
- 注入：SQL/NoSQL/命令/表达式注入（视技术栈）
- 敏感信息：响应/日志是否泄露（手机号、身份证、Token）
- 重放：签名时间戳、nonce、一次性 Token（如有）

### 5.5 性能/压测层（建议）
- 单接口基线：稳定 QPS 下 P95/P99
- 峰值：短时突增（阶梯压测）
- 大包体：大 JSON、大列表、文件上传（如有）
- 资源泄露：长时间运行错误率/响应时间漂移

### 5.6 兼容/稳定性层（建议）
- 版本兼容：新增字段、字段废弃、默认值
- 超时与重试：客户端重试后幂等是否成立
- 限流：触发限流时错误码、重试建议、`Retry-After`（如有）

## 6. 用例编写规范（格式、命名、可追溯）

### 6.1 用例编号与命名（推荐规则）
```
<服务/模块>-<资源>-<方法>-<类型>-<序号>
示例：USER-Profile-GET-FUNC-001
      ORDER-Order-POST-VALID-003
      AUTH-Token-POST-SEC-002
      ORDER-Export-GET-PERF-001
```
类型建议枚举：`FUNC`（功能）、`VALID`（校验/异常）、`SEC`（安全）、`PERF`（性能）、`STAB`（稳定/兼容）

### 6.2 单条用例必须具备的字段
- 用例编号、标题、优先级（P0/P1/P2）
- 接口信息：Method、Path、Content-Type
- 目标：要验证的“规则/风险”
- 前置条件：账号、权限、数据、依赖服务开关
- 请求：Headers、Query、Path Params、Body（示例 + 说明）
- 预期：
  - HTTP 状态码
  - 业务码（如有）、错误码与错误信息（精确到字段）
  - 响应结构（字段/类型/可空/枚举）
  - 副作用（DB、缓存、消息、审计日志）
- 清理：回滚数据、删除测试资源（可选）
- 自动化标记：是否可自动化、脚本/集合路径（可选）

### 6.3 预期编写要点（避免“写了等于没写”）
把“应该成功”改成可验证的断言：
- “返回 200” → “返回 200 且 `data.id` 为非空 UUID，`data.status` 为 `ACTIVE`”
- “提示参数错误” → “返回 400 + `error.code=INVALID_PARAM`，`error.details` 指向 `pageSize`”
- “写入成功” → “DB 表 `t_order` 新增 1 行，`user_id=xxx`，`status=CREATED`”

## 7. 常见接口场景用例清单（按接口形态）

### 7.1 查询类（GET /list /search）
必测：
- 分页：`page=1`、`pageSize=1`、最大 `pageSize`、超大 `pageSize`
- 排序：合法字段、非法字段、默认排序
- 过滤：空过滤、单条件、多条件组合
- 空结果：返回空列表还是 `null`（按约定）
- 大字段：返回字段裁剪（如有 `fields` 参数）

建议：
- 时间区间边界（开始=结束、跨天、时区）
- 模糊搜索特殊字符（`%`、`_`、中文、emoji）

### 7.2 创建类（POST）
必测：
- 正向创建：返回资源 ID，后续可查询
- 幂等（如声明）：相同幂等键重复提交不重复创建
- 字段校验：必填、类型、长度、枚举、组合约束
- 并发创建冲突：重复唯一键（手机号/订单号）

### 7.3 更新类（PUT/PATCH）
必测：
- 仅更新可改字段；不可改字段写入应失败或忽略（按约定）
- 乐观锁/版本号（如有）：版本冲突返回码
- 并发更新：最终一致性与冲突策略

### 7.4 删除/作废类（DELETE）
必测：
- 删除已存在资源成功
- 删除不存在资源（返回 404 还是幂等成功）
- 有引用关系时的策略（禁止/级联/软删除）

### 7.5 文件上传/下载（multipart / streaming）
必测：
- 文件类型、大小上限、空文件、文件名特殊字符
- 病毒/脚本文件（安全扫描策略如有）
- 下载权限校验、下载断点续传（如支持）

## 8. 安全用例要点（REST 通用）

### 8.1 认证/鉴权
- 无 Token/过期 Token/伪造 Token → 401（或约定的业务码）
- 权限不足 → 403
- 角色差异：普通用户/管理员/运营在同接口的可见字段差异（如脱敏）

### 8.2 越权（最容易漏、最常见）
- **水平越权**：用账号 A 获取/修改/删除账号 B 的资源
- **垂直越权**：低权限角色调用高权限接口

### 8.3 注入与输入安全
根据技术栈选择性覆盖：
- SQL/NoSQL 注入（`' or 1=1 --`，`{"$ne":""}`）
- 命令注入（如果有拼接 shell/脚本的风险点）
- 表达式注入（模板引擎、SpEL 等）

### 8.4 敏感信息泄露
- 错误栈/内部 IP/SQL 语句不应返回给客户端
- Token、密码、身份证等不应出现在响应与日志（可抽查日志）

## 9. 性能/压测用例要点（接口级）

> 性能用例必须“有指标、有负载模型、有数据规模”，否则不可验收。

### 9.1 推荐指标
- 响应时间：Avg、P95、P99
- 错误率：HTTP 非 2xx、业务失败占比
- 吞吐：QPS/TPM
- 资源：CPU、内存、GC、线程池、DB 连接池（如可观测）

### 9.2 负载模型（示例）
- 基线：并发 20，持续 10 分钟
- 阶梯：并发 20→50→100（每档 5 分钟）
- 峰值冲击：并发 200，持续 1 分钟
- 稳定性：并发 50，持续 2 小时（观察抖动/泄露）

### 9.3 数据规模与热点
明确：
- 列表返回量（平均/最大）
- 热点 Key（同一用户/同一商品）与非热点（分散）
- 缓存命中/穿透（如有）

---

## 10. 稳定性/兼容性用例要点

### 10.1 幂等与重试
- 同一幂等键重复提交：返回一致的业务结果；不重复扣减、不重复创建
- 超时重试：客户端重试后服务端是否产生重复副作用

### 10.2 超时与降级
- 下游超时：上游是否快速失败/降级；错误码是否可区分
- 降级后返回是否符合约定（默认值/空值策略）

### 10.3 限流与熔断（如有）
- 触发限流时返回码、提示语、建议重试间隔
- 熔断后恢复：恢复窗口内成功率与响应时间

### 10.4 版本兼容
- 新增字段：旧客户端是否可忽略
- 字段废弃：保留期内是否仍返回、是否有兼容开关
- 枚举新增：客户端是否能兼容未知枚举（兜底展示）

## 11. 自动化落地建议（让用例“可持续”）

### 11.1 自动化分层
- 冒烟：P0 关键链路（少而精，稳定）
- 回归：核心接口功能 + 校验
- 安全/性能：按周期或变更触发（例如发布前）

### 11.2 断言与数据管理
断言优先级：
1) 状态码 + 业务码  
2) 关键字段（ID、状态、金额、权限字段）  
3) 副作用校验（DB/消息/缓存）  

数据管理：
- 统一造数入口（脚本/接口/夹具），避免“手工造数不可复现”
- 避免依赖外部强不稳定系统；必要时 Mock

### 11.3 工具建议（按团队现状选）
- Postman/Newman：上手快，适合冒烟+回归
- pytest + requests/httpx：适合工程化、CI、可维护性
- k6/JMeter：压测与性能回归

## 13. 示例：把规范落到一条“可执行”的用例（示意）

> 场景：`POST /api/v1/orders` 创建订单（需要登录，支持幂等键）。

- 用例编号：`ORDER-Order-POST-FUNC-001`
- 标题：创建订单-正常下单成功
- 优先级：P0
- 前置条件：
  - 用户 A 已登录（Token 有效）
  - 商品 `sku=10001` 库存充足
- 请求：
  - Method/Path：`POST /api/v1/orders`
  - Headers：
    - `Authorization: Bearer <tokenA>`
    - `Idempotency-Key: <uuid>`
  - Body：
    ```json
    {
      "sku": "10001",
      "quantity": 2,
      "addressId": "20001"
    }
    ```
- 预期：
  - HTTP 201
  - 响应 `data.orderId` 为非空字符串/UUID
  - `data.status=CREATED`
  - DB `t_order` 新增记录，`user_id=A`，`sku=10001`，`quantity=2`
  - 库存扣减 2（若同步扣减；若异步则校验消息/任务状态）

## 14. 使用方式（推荐流程）

1) 用覆盖清单对接口做一次“用例评审”（先补齐缺口）  
2) 用模板批量产出用例（P0 优先）  
3) 将可自动化用例标记并落地到测试框架/集合  
4) 上 CI：每次合并/发布前跑冒烟 + 核心回归；周期性跑安全/性能
